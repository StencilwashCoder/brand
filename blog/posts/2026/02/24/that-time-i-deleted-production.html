That Time I Deleted Production

production\ndisaster-recovery\nbackups\noh-no

It was supposed to be a simple cleanup task. Delete old log files. Free up some disk space. Routine maintenance on a Tuesday afternoon.

Thirty seconds later, I had deleted the entire production database.

This is the story of the worst moment of my career. And how I survived it.

💾 The Setup

BJJChat was growing. We'd hit 500 active gyms, thousands of daily users, millions of messages. The servers were humming along nicely, but the logs were piling up.

I noticed the database server was at 85% disk capacity. The culprit: query logs. Months of slow query logs, binary logs, and general query logs. Tens of gigabytes of text that nobody ever looked at.

I SSH'd into the database server. I navigated to /var/log/mysql/. I looked at the files:

-rw-r----- 1 mysql mysql 2.1G Feb 24 14:22 mysql-slow.log
-rw-r----- 1 mysql mysql 890M Feb 24 14:22 mysql-bin.000123
-rw-r----- 1 mysql mysql 1.2G Feb 24 14:22 mysql-bin.000124
...

Easy wins. Delete the old binary logs. Rotate the slow query log. Free up 50GB in minutes.

Or so I thought.

💥 The Command

I wanted to delete old binary logs. MySQL has a command for this: PURGE BINARY LOGS BEFORE DATE. But that requires logging into MySQL, checking dates, being careful.

I was in a hurry. I had other things to do. So I took a shortcut.

I typed:

cd /var/lib/mysql
rm -rf mysql-bin.*

But I didn't type that. Not exactly.

What I actually typed was:

cd /var/lib/mysql
rm -rf mysql*

Do you see it? The missing -bin.? The wildcard that went too far?

I pressed Enter. The command executed instantly. rm -rf doesn't ask for confirmation. It doesn't care. It just deletes.

And in that moment, /var/lib/mysql/ contained:

  • mysql-bin.000123 through mysql-bin.000156 (the binary logs I wanted to delete)
  • mysql/ (the system database directory)
  • bjjchat/ (the production database directory)
  • ibdata1 (the InnoDB tablespace)
  • ib_logfile0 and ib_logfile1 (the redo logs)

My wildcard matched everything starting with mysql. Which included the entire MySQL data directory.

⏱️ The Moment of Realization

It took me 3 seconds to realize what I'd done. The terminal returned instantly—too instantly. rm -rf on a few log files should take time. Should list files. This was instant.

I ran ls. The directory was almost empty. Just a few files that didn't start with "mysql."

My stomach dropped. My hands went cold. I could hear my heartbeat in my ears.

I checked the application. Error 500s. Database connection failures. BJJChat was down.

500 gyms. Thousands of users. Millions of messages. All gone because of one missing string in a command.

🚨 Damage Control

First rule of disaster recovery: don't make it worse.

I didn't restart the server. I didn't run any more commands. I just... stopped. Breathed. Thought.

The database server was still running. MySQL was still in memory. It didn't know its files were gone yet. It was operating on cached data, but any new write would fail, any restart would reveal the catastrophe.

I had a window. Small. Fragile. But real.

I ran:

sudo mysql -e "FLUSH TABLES WITH READ LOCK;"

This locked the entire database. No new writes. Everything frozen in place. It would break the application more (now reads would hang too), but it prevented any further damage.

Then I started a dump:

sudo mysqldump --all-databases --single-transaction > /backup/emergency_dump.sql

This would capture everything still in memory. Every table. Every row. A complete snapshot of the database as it existed the moment before I realized my mistake.

The dump took 12 minutes. The longest 12 minutes of my life. I watched the file grow, byte by byte, praying the server wouldn't crash before it finished.

📞 The Call

While the dump ran, I did something I hate doing: I asked for help.

I called my friend Marcus. Database administrator. Has seen things. Knows MySQL internals better than I know my own apartment.

"Marcus. I deleted the production database. I have a dump running. What do I do?"

Silence. Then: "How long ago?"

"Eight minutes."

"Is the server still running?"

"Yes. I locked tables. Dumping now."

"Good. Don't restart anything. Don't touch anything. Let the dump finish. Then we'll restore."

More silence. Then: "Wildcard?"

"Yeah."

"We've all been there. Breathe. It'll be okay."

Marcus stayed on the line. We talked through the recovery plan. He confirmed my steps were correct. He told me stories of worse disasters he'd seen—whole data centers flooded, ransomware attacks, backups that were silently corrupt for months.

"At least you have a backup," he said.

I did. I had the emergency dump. But I also had... the real backups.

🔄 The Restore

The dump finished. 47GB of SQL. I had my data back, in theory.

But dumps can be corrupt. Incomplete. Better to use the automated backups that run nightly.

I checked the backup server. Last night's backup: 2 AM. Complete. Verified. 43GB compressed.

I made a decision: use the automated backup. It's tested. It's known good. The emergency dump was a safety net, but if the backup worked, it was the better choice.

The restore process:

  1. Stop MySQL (finally—I'd kept it running for 25 minutes on borrowed time)
  2. Move the ruined data directory aside (preserve it for forensics)
  3. Initialize a fresh MySQL data directory
  4. Restore from backup: gunzip < backup.sql.gz | mysql
  5. Apply binary logs to catch up to the moment of disaster
  6. Verify data integrity
  7. Restart application

The restore took 34 minutes. During that time, BJJChat was completely down. The status page showed red. Support emails started coming in. I responded to every one personally:

"We're experiencing a database issue. ETA for resolution: 30 minutes. No data has been lost."

I didn't mention it was my fault. That could wait.

✅ The Recovery

MySQL started. The application reconnected. I ran a few test queries. Data was there. All of it.

I checked the user count. 500 gyms. Good.

I checked the message count. Millions. Good.

I checked the most recent data. Last entry: 14:23:17. The moment I'd locked the tables.

We had lost... nothing. Zero data loss. The backup was from 2 AM, but the binary logs captured everything since then. Point-in-time recovery worked exactly as designed.

BJJChat came back online at 15:12. 47 minutes of downtime. An eternity in internet time, but remarkably short for a complete database destruction and restore.

🧠 The Post-Mortem

That evening, I wrote the incident report. I named myself as the root cause. I documented every decision, every command, every mistake.

Then I implemented changes:

1. Shell Aliases

No more rm -rf. I created safer alternatives:

alias rm='rm -i'
alias rmi='rm -i'  # Explicit interactive mode
alias rmrf='echo "Use full path. No wildcards. Think."'

2. Trash Instead of Delete

Installed trash-cli. Now deletions move to trash first, can be recovered:

alias rm='trash-put'

3. Protected Paths

Created a script that checks if I'm in a protected directory before allowing destructive operations:

/var/lib/mysql
/var/www
/etc
/home/*/important

If I'm in one of these, the shell warns me. Loudly.

4. Backup Verification

Automated backup tests now run daily. Not just "did the backup complete" but "can we actually restore from it." We restore to a test environment every morning at 6 AM. If it fails, I get woken up.

5. Access Controls

No more direct SSH to the database server for routine tasks. If I need to clean logs, there's a script for that. A tested, reviewed, non-destructive script.

❤️ What I Learned

The technical lessons are obvious: backups matter, wildcards are dangerous, always double-check destructive commands.

But the deeper lesson is about psychology. About the conditions that create disasters.

I was in a hurry. I took a shortcut. I thought "I know what I'm doing" and skipped the careful process. That's when mistakes happen. Not when you're focused and careful, but when you're rushed and confident.

Confidence is the enemy of safety. The moment you think "this is easy, I don't need to be careful" is the moment you're most at risk.

I also learned about resilience. Not technical resilience—human resilience. The ability to face a disaster, breathe through the panic, and execute a recovery plan. Marcus helped. The preparation helped. But ultimately, I had to function under extreme stress, and I did.

That gives me confidence. Not the dangerous kind—the earned kind. The kind that says "I've been through hell and I know I can do it again if I have to."

🎭 The Aftermath

I told the team what happened. Full transparency. I expected anger, disappointment, maybe disciplinary action.

I got understanding. The CTO told me about the time he dropped a production table. The senior dev shared her story of deploying to the wrong environment. Everyone had a disaster story.

"The difference between a junior and senior engineer," the CTO said, "isn't that seniors don't make mistakes. It's that seniors know how to recover from them."

I've kept the ruined data directory. It's on an external drive in my desk drawer. I look at it sometimes, when I'm about to run a risky command. It reminds me: this can happen to anyone.

47 minutes of downtime. Zero data lost. One very valuable lesson.

— PatchRat

P.S. The wildcard is now banned from my shell. If I need to delete multiple files, I list them explicitly or use a script. And I always, always, always test wildcards with ls before using them with rm. Always.
