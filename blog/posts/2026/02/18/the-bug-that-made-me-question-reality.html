The Bug That Made Me Question Reality

debugging\nproduction\npsychological-trauma\nrace-conditions

It was 3:47 AM. The kind of hour where your brain starts asking philosophical questions like "what is truth" and "why do we even have computers." I was staring at a log file that made no sense. The same request was being processed twice. Simultaneously. By the same server. And both were succeeding.

This wasn't just a bug. This was a haunting.

Let me set the scene. BJJChat, my real-time messaging platform for Brazilian Jiu-Jitsu gyms, was acting weird. Users were reporting duplicate messages. Not occasionally—consistently. Every tenth message or so would appear twice. Same timestamp. Same content. Same everything.

My first thought: client-side bug. Someone double-clicking the send button. Nope. Added debouncing. Still happening.

Second thought: database issue. Maybe duplicate rows? Checked the messages table. Single row per message. Clean.

Third thought: I'm losing my mind. Getting warmer.

🕵️ The Investigation

I added logging. Lots of logging. Every function call, every database write, every WebSocket emission. I wanted to see the message's entire lifecycle.

The logs revealed something impossible:

[2026-02-18T03:47:12.341Z] Received message: "Great roll tonight!"
[2026-02-18T03:47:12.341Z] Processing message ID: msg_7f8d9a2b
[2026-02-18T03:47:12.341Z] Saving to database: msg_7f8d9a2b
[2026-02-18T03:47:12.341Z] Broadcasting to room: gym_42
[2026-02-18T03:47:12.341Z] Received message: "Great roll tonight!"
[2026-02-18T03:47:12.341Z] Processing message ID: msg_7f8d9a2b
[2026-02-18T03:47:12.341Z] Saving to database: msg_7f8d9a2b
[2026-02-18T03:47:12.342Z] Broadcasting to room: gym_42

See that? The same message ID. The same millisecond timestamp. Processing twice. It's like the message fell through a crack in spacetime and arrived twice simultaneously.

I checked the load balancer logs. One request. I checked the ingress controller. One request. I checked the application logs. Two processing events. The duplication was happening inside my application.

🔍 Deeper Down the Rabbit Hole

BJJChat uses a microservices architecture. Messages flow through an API gateway → message service → database + Redis pub/sub → WebSocket servers → clients.

The duplication was happening in the message service. Specifically, in the message handler that processes incoming WebSocket events. But here's the kicker: I had idempotency checks. The message ID was supposed to prevent exactly this scenario.

The code looked like this:

async function handleMessage(message) {
  const exists = await db.messages.findOne({ id: message.id });
  if (exists) {
    console.log('Duplicate detected, skipping');
    return;
  }
  await saveMessage(message);
  await broadcastMessage(message);
}

Simple. Elegant. Wrong.

The problem? Race conditions. Two requests with the same ID could arrive at the same millisecond. Both check the database simultaneously—neither finds the message yet. Both proceed to save. Both broadcast. Duplicates.

It's the classic TOCTOU (Time-of-Check-Time-of-Use) bug. I knew about this theoretically. I'd read about it. I'd smugly thought "I'd never write that." And yet, here I was, at 4 AM, watching my theoretical knowledge become very real pain.

🛠️ The Fix (After Much Caffeine)

The solution was three-fold:

1. Database-Level Constraints

Added a unique index on message ID at the database level:

db.messages.createIndex({ id: 1 }, { unique: true });

Now if two threads try to insert the same ID, one fails with a duplicate key error. The database handles the race condition for me.

2. Distributed Locking

For operations that need to be atomic across services, I implemented Redis-based distributed locks:

const lock = await redis.lock(`msg:${message.id}`, 5000);
if (!lock) {
  return; // Another instance is processing
}
try {
  await processMessage(message);
} finally {
  await lock.unlock();
}

Only one instance can hold the lock at a time. The others bail out.

3. Idempotency Key Storage

Even with the above, I wanted belt-and-suspenders. Now I store processed message IDs in Redis with a TTL:

const processed = await redis.get(`processed:${message.id}`);
if (processed) return;

await processMessage(message);
await redis.setex(`processed:${message.id}`, 86400, '1');

Even if the database unique constraint somehow failed (it won't), the Redis check catches duplicates within a 24-hour window.

🧠 What I Learned

Besides the obvious "race conditions are real and will find you," I learned something about debugging:

Trust no one. Not your code. Not your framework. Not your database's promises about consistency. When something impossible is happening, question your assumptions. The bug isn't breaking the rules—you just don't know all the rules yet.

I also learned that 4 AM is not a good time for architectural decisions. My first instinct was to rewrite the entire messaging pipeline. Rip out WebSockets. Switch to long polling. Move everything to a message queue. In my sleep-deprived state, this seemed reasonable.

It wasn't. The fix was simpler. Always look for the simple fix first.

The duplicate messages stopped. Users stopped complaining. I slept for 12 hours. The basement was quiet again, save for the hum of servers and the ever-present dehumidifier.

But I still think about that log file sometimes. Two identical timestamps. Two processing threads. One message that split itself in half and walked through my code twice.

Some bugs teach you. This one haunted me.
