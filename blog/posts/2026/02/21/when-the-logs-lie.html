When the Logs Lie

debugging\nlogging\nobservability\ntrust-no-one

The logs said everything was fine. No errors. No warnings. Just the gentle, rhythmic heartbeat of a healthy application—requests coming in, responses going out, business as usual.

Except users couldn't log in.

Not all users. Just... some users. Randomly. Intermittently. The worst kind of bug. The kind that makes you question your sanity, your code, and whether computers are fundamentally evil.

This is the story of the day I learned that logs can lie.

🔍 The Symptoms

It started with a Slack message: "Hey, can't log into the dashboard. Just sits there spinning."

I tried it myself. Worked fine. Logged in. Logged out. No problems.

"Must be a cache issue," I said, confidently. "Try clearing cookies."

They did. It didn't help.

Then another message: "Same issue here." Then another. Soon we had five people affected, then ten. But I still couldn't reproduce it. My account worked perfectly.

I checked the logs:

[2026-02-21T09:15:23Z] POST /api/auth/login - 200 OK - 145ms
[2026-02-21T09:15:24Z] POST /api/auth/login - 200 OK - 132ms
[2026-02-21T09:15:25Z] POST /api/auth/login - 200 OK - 128ms

All green. All 200s. Fast response times. Nothing to see here.

But the users weren't lying. They showed me screenshots. Login page. Credentials entered. Submit button clicked. Loading spinner... forever.

Something was very wrong.

🕵️ Deep Dive

Since I couldn't reproduce the issue with my account, I asked a user to help me debug. They graciously agreed (bless patient users). I added their user ID to a special debug list that would log everything.

Their logs looked identical to mine:

[2026-02-21T10:42:11Z] POST /api/auth/login - 200 OK - 156ms
[2026-02-21T10:42:11Z] Generated JWT token
[2026-02-21T10:42:11Z] Set-Cookie: auth_token=eyJhbGci...

Login successful. Token generated. Cookie set. And yet, they couldn't access the dashboard.

I added more logging. I logged the token payload. I logged the cookie settings. I logged the redirect URL. Everything looked perfect. The server was doing exactly what it was supposed to do.

So why couldn't they log in?

🎯 The Breakthrough

After two hours of adding logs and finding nothing, I did something desperate: I looked at the browser's developer tools.

Not the console. The Network tab. Specifically, the Cookies section.

There it was. The smoking gun.

The server was sending the cookie. The browser was rejecting it. A tiny yellow warning icon in Chrome's dev tools: "This cookie was blocked due to user preferences."

I stared at that warning for a solid minute. User preferences? What did that mean?

Then I remembered: third-party cookie blocking.

💡 The Realization

Here's what was happening:

The application had recently moved to a new domain structure. The API was on api.app.com. The frontend was on app.com. This worked fine—cookies were set with the .app.com domain, making them available to both.

But Chrome had tightened its third-party cookie policies. And here's the thing: api.app.com and app.com are technically cross-origin. They're subdomains, but they're different origins. Chrome was treating the API's cookie as a third-party cookie.

For most users, this was fine. Chrome's third-party cookie blocking has exceptions for SameSite=None cookies with Secure and HttpOnly flags. Which we had. But some users—those with stricter privacy settings, or certain enterprise policies—had complete third-party cookie blocking enabled.

The server sent the cookie. The browser said "nah." The user couldn't authenticate on subsequent requests because no cookie was being sent.

And here's the kicker: my browser didn't have this restriction. That's why I couldn't reproduce it. I had normal privacy settings. I saw the cookie being set in my browser. Everything worked.

The logs were telling the truth about the server. The server was telling the truth about the response. But somewhere between server and user, the browser was lying by omission. Not rejecting the cookie visibly. Just... silently ignoring it.

🛠️ The Fix

The immediate fix was to move the API back to the same origin: app.com/api instead of api.app.com. This eliminated the cross-origin issue entirely. Cookies were now first-party by definition.

But that was a band-aid. The real fix required understanding and implementing proper cross-origin authentication:

Option 1: Same Origin

Keep API and frontend on the same domain. No cross-origin issues, no third-party cookie problems. Simple. Effective. Boring.

Option 2: Token-Based Auth

Instead of cookies, use JWT tokens in the Authorization header. Store them in memory (or localStorage with appropriate XSS protections). No cookies, no cookie policies.

Option 3: Proper CORS and SameSite

If you must be cross-origin, do it right:

Set-Cookie: auth_token=xxx; SameSite=None; Secure; HttpOnly; Partitioned;

The Partitioned attribute is key. It enables CHIPS (Cookies Having Independent Partitioned State), which allows cross-origin cookies in a privacy-preserving way.

We went with Option 1 for the immediate fix, and started planning Option 2 for the long term. Token-based auth is more complex but avoids all the cookie nonsense entirely.

🧠 Lessons Learned

This bug taught me several things:

1. Logs Don't Show the Whole Picture

Server logs show what the server did. They don't show what happened after the bytes left the network interface. The client can reject, modify, or ignore anything. Trust your logs, but verify with client-side evidence.

2. Reproduction is Everything

If you can't reproduce a bug, you're missing context. The user's environment differs from yours in ways you haven't identified. Browser version, extensions, privacy settings, network conditions—all can affect behavior.

3. Cookie Policies are Moving Targets

Browser vendors are constantly tightening privacy. What worked yesterday might break tomorrow. Stay current with cookie specifications. Test with strict privacy settings. Assume cookies will be blocked and have a fallback.

4. The Network Tab is Your Friend

I should have looked there first. Hours of log analysis revealed nothing that 30 seconds in the Network tab couldn't have shown. When dealing with client-server issues, verify both sides.

🔮 Prevention

How do we prevent this in the future?

First, we added monitoring that actually checks client-side behavior. We have a synthetic test that logs in using a headless browser with strict privacy settings. If it fails, we know immediately.

Second, we documented the cookie requirements. Every developer knows now: if you touch authentication, test with third-party cookies blocked.

Third, we added client-side error reporting. If a login succeeds on the server but the user can't access the app, we capture that. We send diagnostics. We know something went wrong even if the logs look clean.

🎭 The Aftermath

The affected users could log in again within an hour of the fix. Most never knew what happened—they just tried again later and it worked.

But I knew. I knew that for five hours, my application was broken for a subset of users, and my logs told me everything was fine. I knew that I had trusted my observability without questioning its blind spots.

Logs are a map. But maps aren't the territory. Sometimes you have to walk the ground yourself.
