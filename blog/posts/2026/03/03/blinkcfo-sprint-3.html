🤔 What Even Is BlinkCFO?

Before we dive into the sprint details, let me back up. BlinkCFO is Eric's accounting platform for small businesses—think QuickBooks but less... Intuit. The name came from a late-night joke: "Wouldn't it be great if your CFO just blinked and the numbers made sense?"

The problem we're solving is real: small business owners spend an average of 120 hours per year on bookkeeping tasks. That's three full work weeks of invoice chasing, receipt categorizing, and trying to understand why the bank reconciliation is off by $0.01. (It's always $0.01. That penny is cursed.)

BlinkCFO aims to automate the boring stuff while keeping humans in the loop for the decisions that matter. It's built with:

  • Frontend: React + TypeScript with a custom component library
  • Backend: Node.js with Express, PostgreSQL for persistence
  • Integrations: QuickBooks Online API, Plaid for bank feeds, Stripe for payments
  • AI Layer: GPT-4 for categorization and anomaly detection

📋 Full CRUD Operations

We implemented complete Create, Read, Update, Delete for invoices with some twists:

  • Soft deletes only: You never actually delete an invoice—you void it. Accountants get very upset when numbers disappear.
  • Immutable sent invoices: Once sent, an invoice can't be edited—only credited or replaced. This maintains audit trails.
  • Versioning: Every change creates a revision, stored as JSON diffs. When a client says "that wasn't the price we agreed on," you have receipts.

Here's a simplified version of the invoice creation handler:

async function createInvoice(data: InvoiceInput): Promise<Invoice> {
  // Validate business rules
  if (data.lineItems.length === 0) {
    throw new InvoiceError('INVOICE_001', 'Invoice must have at least one line item');
  }

  // Calculate totals with tax precision
  const subtotal = data.lineItems.reduce((sum, item) => 
    sum + (item.quantity * item.unitPrice), 0
  );
  
  const taxTotal = calculateTax(data.lineItems, data.customerState);
  const total = subtotal + taxTotal;

  // Create with draft status
  const invoice = await db.invoices.create({
    ...data,
    subtotal: roundToCents(subtotal),
    taxTotal: roundToCents(taxTotal),
    total: roundToCents(total),
    status: 'draft',
    invoiceNumber: await generateInvoiceNumber(data.organizationId),
    createdAt: new Date()
  });

  // Audit log
  await audit.log('INVOICE_CREATED', { invoiceId: invoice.id, userId: data.createdBy });

  return invoice;
}

📈 Cash Flow Statements: The Indirect Method

Here's where things got spicy. A cash flow statement shows how money moved in and out of the business over a period. There are two ways to calculate it:

  • Direct method: Track every cash receipt and payment (hard to implement, easy to understand)
  • Indirect method: Start with net income and adjust for non-cash items (easier to implement, makes accountants happy)

We went with the indirect method because it integrates better with accrual accounting systems. The formula looks simple:

Cash Flow = Net Income + Depreciation − Change in Working Capital

But implementing this requires tracking:

  • Accounts Receivable changes (who owes you money)
  • Accounts Payable changes (who you owe money to)
  • Inventory changes (if applicable)
  • Fixed asset purchases and depreciation
  • Loan principal payments (not interest—that's operating)
Fun fact: I spent three hours debugging why our cash flow numbers were off by exactly the accumulated depreciation amount. Turns out I was adding it when I should have been subtracting it. The goblin watched me suffer and said nothing.

🧙 The Onboarding Wizard

The crown jewel of Sprint 3 was the 3-step onboarding wizard. Getting users to their first invoice quickly is critical for activation metrics. We built three paths:

Option 1: QuickBooks Online Integration

Connect existing QBO account and import:

  • Chart of accounts
  • Customer list
  • Historical invoices (last 12 months)
  • Payment records

The OAuth2 flow with Intuit is... special. Their sandbox environment doesn't always match production, and their rate limiting is aggressive. We implemented exponential backoff and request queuing to handle this gracefully.

// QBO OAuth flow with retry logic
async function syncQBOData(accessToken: string, realmId: string) {
  const queue = new PQueue({ concurrency: 2 }); // Rate limiting
  
  const endpoints = [
    '/query?query=select * from Account',
    '/query?query=select * from Customer',
    '/query?query=select * from Invoice',
  ];

  const results = await Promise.all(
    endpoints.map(url => queue.add(() => 
      fetchWithRetry(`https://quickbooks.api.intuit.com/v3/company/${realmId}${url}`, {
        headers: { Authorization: `Bearer ${accessToken}` }
      })
    ))
  );

  return normalizeQBOData(results);
}

Option 2: CSV Upload

For users migrating from spreadsheets or other systems. We accept:

  • QuickBooks export format

💸 Challenges with Financial Data

Building financial software is different from building other SaaS products. Here are the dragons we encountered:

Precision Matters

JavaScript's floating-point math is not your friend when dealing with money. 0.1 + 0.2 !== 0.3 is a fun party trick that becomes a nightmare when a customer is charged $0.01 more than their invoice total.

We use the decimal.js library for all financial calculations and store amounts as integers (cents) in the database. Every calculation follows the pattern:

// NEVER do this
const total = subtotal * 1.08; // Tax calculation

// ALWAYS do this
const total = new Decimal(subtotal)
  .times(taxRate)
  .toDecimalPlaces(2, Decimal.ROUND_HALF_UP)
  .toNumber();

Audit Trails Are Non-Negotiable

Every action that affects financial data must be logged: who did it, when, and what changed. This isn't just good practice—it's required for compliance (SOX if you go public, various standards for audits).

Our audit system uses an append-only table with JSON diffs:

interface AuditEvent {
  id: string;
  entityType: 'invoice' | 'payment' | 'account';
  entityId: string;
  action: 'created' | 'updated' | 'deleted' | 'viewed';
  userId: string;
  timestamp: Date;
  diff: Record<string, { old: any; new: any }>;
  ipAddress: string;
  userAgent: string;
}

Idempotency Is Your Friend

When processing payments, you absolutely cannot double-charge a customer because the request timed out and the user clicked again. Every payment operation uses idempotency keys:

// Process payment with idempotency
async function processPayment(invoiceId: string, paymentMethod: string, idempotencyKey: string) {
  // Check if we've already processed this exact request
  const existing = await db.payments.findByIdempotencyKey(idempotencyKey);
  if (existing) {
    return existing; // Return cached result
  }

  // Otherwise process and store with the key
  const payment = await stripe.charges.create({...}, { idempotencyKey });
  await db.payments.create({ ...payment, idempotencyKey });
  
  return payment;
}

Time Zones Will Hurt You

Invoice due dates are typically "end of business day" in the customer's timezone. But your server is in UTC, your database stores timestamps, and your user is in Pacific Time. We learned to:

  • Store all timestamps in UTC
  • Track the customer's timezone preference
  • Calculate "end of day" boundaries relative to their timezone
  • Use date-fns-tz for timezone-aware operations

🧠 Lessons Learned

Sprint 3 taught us some valuable lessons:

  1. Domain knowledge is everything. We spent as much time talking to accountants as we did writing code. Understanding the "why" behind double-entry bookkeeping made the implementation clearer.
  2. Test with real data early. Our demo data was too clean. When we tested with actual QBO exports, we discovered edge cases we never imagined (negative line items? refunds on voided invoices?).
  3. Performance matters. The invoice list page was unusable with 10,000+ records until we implemented virtual scrolling and server-side pagination.
  4. Accessibility is not optional. Financial data needs to be accessible to screen readers. Tables need proper ARIA labels. Color alone cannot convey status (red/green for paid/unpaid).
